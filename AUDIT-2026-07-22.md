# health_monitor — Audit de code

Scope : gem complet — 16 fichiers de production (`lib/**`, `app/views/**`, `config/routes.rb`, `extra/check_rails.rb`) + surface projet (gemspec, Gemfile, Appraisals, CI, RuboCop, README) · specs auditées en méta (couverture, CI). · Couverture : 16/16 fichiers de production lus.
Sémantique vérifiée ce tour :
- `raise SidekiqException, e` avec `e` objet exception → le message est préservé en `String` (`"original boom"`), pas de dégradation → **non-bug**.
- `[].any? { |x| x != 'OK' }` → `false` → confirme que zéro provider agrège en statut `:ok` (200).

Dimensionnement : ~16 fichiers de production + 33 specs → très en-dessous du seuil « quelques centaines » ; couverture exhaustive honnête et réalisée en une passe.

## Résumé priorisé (le plus grave d'abord)

1. **MEDIUM — COR — filtrage par un provider inconnu/non activé → 200 `ok` avec zéro contrôle** (« vert silencieux »). Le cœur du problème : un monitoring mal configuré croit le service sain alors que rien n'a été vérifié.
2. **MEDIUM — SEC — endpoint non authentifié par défaut expose les messages d'exception bruts** (chemins internes, hôtes, potentiellement une URL Redis avec credentials).
3. **MEDIUM — COR/ressource — le provider Redis ferme une connexion qu'il ne possède pas** (connexion injectée ou `ConnectionPool`).
4. **MEDIUM — CON — clé de health-check partagée (`health:<ip>` / `health:`) → course concurrente → faux `ERROR` 503** (flapping du monitoring).
5. **LOW — API — un `ServiceWarning` produit un 503 identique à une erreur** (la distinction WARNING/ERROR est cosmétique au niveau HTTP).
6. **LOW — DOC — Delayed Job documenté (réglages `queue_size`) mais non activable via `config.delayed_job`** (absent de `PROVIDERS`).

## 1. Correctness

```
MEDIUM | COR-health_monitor.rb:32 | lib/health_monitor.rb:32-42 | Un paramètre `providers[]` ne matchant aucun provider enregistré produit une liste de résultats vide, agrégée en `status: :ok` (HTTP 200). | Un monitoring interroge `/check.json?providers[]=postgres` (ou une faute de frappe `databse`, ou un provider non activé sur cet hôte) → `results: []`, `status: "ok"`, 200 OK → une panne réelle de la base passe totalement inaperçue. Comportement confirmé et figé par le test `spec/controllers/.../health_controller_spec.rb:88` (« returns empty providers » avec `status: ok`). | Si `params[:providers]` est fourni mais que l'intersection avec les providers connus est vide, renvoyer une erreur (`:service_unavailable` / 422) plutôt qu'un OK vide ; alternativement, valider les noms demandés contre `configuration.providers` et signaler les inconnus.
```

```
MEDIUM | COR-redis.rb:44 | lib/health_monitor/providers/redis.rb:44-52,79-85 | `redis_close` ferme inconditionnellement `@redis`, y compris lorsque la connexion a été *injectée* (`configuration.connection`) ou est un `ConnectionPool` — objets que le provider ne possède pas. | `config.redis.connection = $shared_redis` : chaque appel à `/check` exécute `@redis.close` sur la connexion partagée ; du code applicatif l'utilisant en concurrence récupère une connexion fermée (reconnexion forcée, ou erreur sous charge). ## 2. Concurrency
```

```
MEDIUM | CON-cache.rb:11 | lib/health_monitor/providers/cache.rb:11,17-20 | La clé de sonde `health:<remote_ip>` (ou `health:` quand `request` est nil) est partagée entre requêtes de même IP ; le motif write-puis-read n'est pas atomique. | Deux sondes concurrentes derrière la même IP (LB, probes k8s d'un même nœud, NAT) : requête A écrit tA, requête B écrit tB, A relit tB → `tA != tB` → `CacheException` → faux `ERROR` / 503 → alerte/flapping non fondés. Même schéma sur `HealthMonitor.check` sans `request` (clé `health:` globale à toutes les sondes). | Rendre la clé unique par sonde (SecureRandom, ou inclure un jeton par requête/thread) et la nettoyer, ou remplacer le round-trip write/read par un contrôle sans état partagé (`Rails.cache.read`/increment sur clé unique).
```

```
MEDIUM | CON-redis.rb:30 | lib/health_monitor/providers/redis.rb:30,54-60 | Même défaut que le cache : `@key = ['health', remote_ip].join(':')` puis `set` immédiatement suivi de `get` comparé à la valeur écrite. | Sondes Redis concurrentes de même IP : la valeur relue est celle écrite par l'autre sonde → « different values » → `RedisException` → faux 503. | Clé unique par sonde (voir CON-cache.rb:11).
```

## 3. Error model

```
LOW | ERR-redis.rb:36 | lib/health_monitor/providers/redis.rb:36-40 | Le `ensure redis_close` s'exécute après un échec de `check_values!` sur une connexion potentiellement déjà cassée ; une exception levée par `close` masquerait l'erreur d'origine. | Redis tombe pendant `check_values!` (lève) → `ensure` appelle `redis_close` → si `close` lève à son tour, l'exception de fermeture remonte à la place du vrai motif, brouillant le message affiché. | Envelopper `redis_close` dans un `rescue nil` (ou logguer), ou ne fermer que si la connexion est vivante.
```

Reste de la dimension : les providers rescue-then-reraise en `*Exception < ServiceError` de façon cohérente ; `provider_result` (health_monitor.rb:47-72) capture `ServiceWarning` puis `Error, StandardError` séparément — correct. Pas d'exit-code trompeur : le statut HTTP dérive de l'agrégat (hors cas COR-health_monitor.rb:32).

## 4. Performance / memory / CPU

Aucun défaut significatif. Le nombre de providers est petit et fixé à la configuration ; pas de N+1, pas de chargement non borné. `DelayedJob#check_queue_size!` fait un `COUNT` unique. Le commit récent 65bc846 a déjà supprimé l'appel triple des checks par requête. **Rien à signaler.**

## 5. API design

```
LOW | API-health_monitor.rb:40 | lib/health_monitor.rb:40 | Un `ServiceWarning` (statut `WARNING`) agrège en `:service_unavailable` (503), exactement comme une erreur — la distinction WARNING/ERROR n'existe qu'au niveau du corps, pas du code HTTP. | Un provider custom lève `Error::ServiceWarning` en pensant « dégradé mais up » ; le load-balancer reçoit 503 et sort le nœud de rotation comme pour une panne dure. | Décider explicitement de la sémantique : soit documenter que WARNING = 503, soit ne passer en 503 que si un résultat est `ERROR` (`results.any? { |r| r[:status] == STATUSES[:error] }`).
```

Note (design, non-bug) : `Configuration` expose les providers built-in par métaprogrammation (`class_eval`) — lisible et testé ; le contrat `Base` (`check!` + `configuration_class` optionnelle) est propre.

## 6. Security

```
MEDIUM | SEC-health_controller.rb:34 | lib/health_monitor/health_controller.rb:34-41 + lib/health_monitor.rb:53,62,70 | L'endpoint `/check` n'est pas authentifié par défaut (`basic_auth_credentials` nil) et le corps renvoie `e.message` brut de chaque provider en échec. | Un attaquant non authentifié interroge `/check.json` pendant une panne : les messages d'exception peuvent divulguer hôtes/chemins internes, et notamment — si `redis_config.url = 'redis://user:pass@host'` — une erreur de connexion redis-rb incluant l'URL avec credentials. Divulgation d'information / potentiellement de secret, silencieuse côté opérateur. | Recommander fortement (README + éventuel garde-fou) l'activation de `basic_auth_credentials` ; en complément, assainir/tronquer les messages exposés (ne pas propager le message brut du backend dans la réponse HTTP publique).
```

Vérifié sans défaut : la vue ERB `check.html.erb` interpole `status[:name]/[:status]/[:message]` via `<%= %>` (auto-échappé par Rails) → pas de XSS même si un message d'erreur contient du HTML. `providers_params` utilise `permit` (pas de `permit!`). Route `GET` seule → CSRF non pertinent. `extra/check_rails.rb` : `URI(options[:uri])` + `basic_auth` — l'URI vient de la ligne de commande de l'opérateur (pas d'entrée non fiable), pas d'injection shell.

## Hypothèses vérifiées et écartées

- **`raise SidekiqException, e` (sidekiq.rb:44) passerait un objet Exception comme message → message dégradé.** Écarté : probe Ruby ce tour → `message == "original boom"` (String). Ruby appelle `.exception`, le message d'origine est conservé. Non-bug.
- **XSS stocké via `status[:message]` dans la vue HTML.** Écarté : `check.html.erb` utilise `<%= %>` (auto-échappement Rails) ; un message contenant `<script>` est échappé. Confirmé par lecture de la vue.
- **Injection dans `extra/check_rails.rb`.** Écarté : l'URI et les credentials proviennent des arguments CLI de l'opérateur (source fiable), pas d'entrée réseau ; `Net::HTTP` sans exec/shell.
- **`params[:providers]` non typé atteignant une requête SQL (injection / type confusion).** Écarté : `providers_params` fait `permit(providers: [])` et la valeur ne sert qu'à un `include?` en mémoire sur des noms de providers ; jamais interpolée dans une requête.
- **Matrice CI Ruby `4.0` inexistante → CI rouge.** Écarté comme finding : `fail-fast: false` isole la branche, et la disponibilité exacte de Ruby 4.0 ne peut être affirmée sans vérification en ligne (hors périmètre) ; simple observation, pas un défaut du code.
- **État global partagé non thread-safe (`@global_configuration`, `configuration`).** Écarté : écrit une seule fois au boot (`Engine#before_initialize`), lu en lecture seule par requête ; providers instanciés par requête. Pas de mutation concurrente.
